HelperKit — Real-Time AI Transcription App

helperkit.live  ·  Node.js · Express · WebSocket · React · Firebase · AWS EC2 · Deepgram · OpenAI

Node.jsWebSocketAI APIsAWS

What it is: A web app for live speech-to-text captioning during interviews or lectures. Multiple viewers can watch captions in real time, and at the end of a session they get an AI-generated summary. Achieves sub-200ms transcription latency.

System Architecture

[Browser (Speaker)] | MediaRecorder API captures mic audio | WebSocket connection (ws://) v [Node.js + Express Server — AWS EC2] | Receives raw audio chunks | Streams audio to Deepgram STT API | Deepgram returns transcript text | Writes text → Firebase Realtime DB | After session: sends full transcript → OpenAI API → summary v [Firebase Realtime DB] | Syncs in real-time to all connected viewers v [Browser (Viewers)] React.js frontend — listens to Firebase, renders captions live as they arrive [nginx ←→ Node.js] (reverse proxy, port 80/443) [PM2] (process manager, auto-restart) [Let's Encrypt SSL] (HTTPS cert, auto-renew)

End-to-End Flow — Step by Step

1
User opens helperkit.live — nginx receives HTTPS request, proxies to Node.js on internal port. React app loads in browser.
2
Speaker starts session — browser requests mic access via getUserMedia(). MediaRecorder API starts capturing audio in small chunks (e.g. 250ms intervals).
3
WebSocket connection opens — React client opens a persistent WebSocket connection to the Node.js server. This keeps a live bidirectional pipe open — no HTTP request overhead per audio chunk.
4
Audio streams to server — each audio chunk is sent as binary data over the WebSocket. Server receives chunks and immediately forwards them to the Deepgram STT API via Deepgram's streaming WebSocket.
5
Deepgram transcribes — Deepgram's ML models convert speech → text in near-real-time. Returns JSON with transcription text and confidence scores. This is where the sub-200ms latency comes from — streaming, not batch.
6
Text written to Firebase — server pushes each transcript segment to Firebase Realtime Database. Firebase pushes the update to all connected viewer clients instantly via its own WebSocket layer.
7
Viewers see live captions — React frontend listens on Firebase path with onValue(). When new text arrives, it updates the DOM. Works for unlimited concurrent viewers with no extra server load (Firebase handles fan-out).
8
Session ends → AI summary — Node.js collects the full session transcript, sends it to OpenAI API (e.g. GPT-4) with a summarization prompt. Returns a structured summary displayed to the user.

Security

  • HTTPS/SSL via Let's Encrypt — all traffic encrypted
  • nginx reverse proxy — Node.js port never exposed publicly
  • Firebase security rules — controls who can read/write sessions
  • API keys (Deepgram, OpenAI) stored as server env vars, never in frontend
  • PM2 ensures process restarts on crash — no downtime window

Why These Tools?

  • WebSocket over HTTP polling — 10–50× lower latency for streaming audio
  • Deepgram over Whisper — streaming API, lower latency than batch models
  • Firebase for viewer sync — zero backend code needed for fan-out to many viewers
  • React — reactive UI updates naturally when Firebase data changes
  • PM2 — production process manager, handles restarts and logs
💬 Recruiter talking point: "The key engineering challenge was keeping latency under 200ms while supporting concurrent users. WebSocket streaming to Deepgram avoids batch-processing delays, and Firebase handles viewer fan-out so my server isn't the bottleneck when many people watch simultaneously."

2

3

  • Django ORM — parameterized queries prevent SQL injection by default
  • CORS policy — only Angular origin is whitelisted

  • Foreign keys enforce ownership — user X can't read user Y's jobs

🤟 Sign Language Detection & Recognition IEEE PublishedML

CNN-based Gesture Recognition from Live Video

Python · PyTorch · OpenCV · C++ · PCNN features · Temporal aggregation

PyTorchOpenCVCNNC++

What it is: A gesture recognition system that takes live webcam video and classifies American Sign Language gestures in real time. Published at the 2023 IEEE ICSSCS conference.

ML Pipeline Architecture

[Webcam input] | OpenCV — VideoCapture, frame-by-frame reading v [Preprocessing] | Resize frames to fixed size (e.g. 224×224) | Convert BGR → grayscale / RGB normalization | Hand region extraction (bounding box) v [PCNN Feature Extraction] ← C++ accelerated | Phase Congruency Neural Network | Detects edges/contours invariant to brightness changes | Extracts spatial features that generalize across lighting v [CNN Model (PyTorch)] | Convolutional layers → feature maps | Pooling layers → spatial downsampling | Fully connected layers → class probabilities | Softmax output → gesture label v [Temporal Aggregation] | Combines predictions across N consecutive frames | Smooths out single-frame noise | Outputs stable gesture classification v [Output] | Gesture label displayed on frame via OpenCV overlay

End-to-End Flow

1
Frame captured — cv2.VideoCapture(0) grabs each frame from webcam. OpenCV is a C++ library with Python bindings — fast enough for real-time processing.
2
Preprocessing — frame resized, normalized, hand region isolated. This ensures the model sees consistent input regardless of camera distance or position.
3
PCNN feature extraction — Phase Congruency Neural Network extracts edge/contour features that are invariant to lighting changes. C++ used here for speed since this runs on every frame.
4
CNN inference — extracted features passed through the PyTorch CNN model. Convolutional layers detect low-level patterns (edges, curves), deeper layers detect high-level patterns (finger positions, hand shapes). torch.no_grad() used during inference for speed.
5
Temporal aggregation — predictions from last N frames are pooled (e.g. majority vote or average). Prevents flickering output when a single frame is ambiguous mid-gesture.
6
Result displayed — cv2.putText() overlays the predicted label on the frame. cv2.imshow() renders it in a window at real-time frame rate.

What is PCNN?

  • Phase Congruency — measures where image intensity waves are in phase (edge locations)
  • Unlike gradient-based edges (Sobel), PC is invariant to contrast and brightness
  • Better for real-world video where lighting varies
  • Extracts the geometric structure of a gesture, not its pixel values

Why C++ here?

  • PCNN feature extraction is compute-heavy — runs per-frame
  • Python would create too much latency for real-time video
  • C++ module called from Python via bindings (ctypes or pybind11)
  • PyTorch handles the GPU-accelerated CNN; C++ handles CPU preprocessing
💬 Recruiter talking point: "We used PCNN features specifically because standard CNN inputs are sensitive to lighting — in a real-world sign language scenario, lighting changes constantly. PCNN extracts structural features that stay consistent. The IEEE paper validates our accuracy across those varied conditions."

Django Web App — Profile → Auto-Generated PDF Resume

Python · Django · SQLite · ReportLab · HTML/CSS · React (in-browser)

DjangoReportLabSQLite

What it is: A Django web app where students register, fill in their academic and work profile across multiple sections, and download a formatted PDF resume. Admin can post job notifications with required skills for students to browse.

Architecture

[Browser — HTML/CSS + React (in-browser Babel)] | Multi-tab form: Personal, Education, Skills, Projects, Experience | API calls to Django endpoints v [Django Backend] | Session-based authentication (Django sessions) | Views: registration, login, profile_update, download_pdf | Django ORM → SQLite | PDF generation: ReportLab Platypus SimpleDocTemplate v [SQLite Database] | Tables: UserModel (all profile fields), NotificationModel (job posts) | pic field: FileField — stores uploaded photo to /images/ v [ReportLab PDF Engine] | Fetches user data from DB | Builds PDF: header with photo, sections, skills grid, projects | Returns as FileResponse — browser downloads directly

End-to-End Flow

1
Register — user fills registration form including uploading a photo. Django's FileField saves photo to MEDIA_ROOT/images/. User record created in SQLite via UserModel.save().
2
Login — Django validates credentials, creates a session. request.session['user_id'] set. CSRF token sent with every subsequent form submission.
3
Fill profile — React frontend tabs POST to /api/profile/update/. Django view updates all fields on the UserModel record. Data persists to SQLite via Django ORM.
4
Download PDF — user clicks download → GET /api/download/ → Django fetches UserModel from DB → passes all fields into ReportLab → PDF built in memory using SimpleDocTemplate with Platypus flowables (Paragraphs, Tables, Images) → returned as FileResponse.
5
Admin notifications — admin logs into Django admin panel → posts job notifications with required skills → students see them in their dashboard as job alerts.

What is ReportLab?

  • Python PDF generation library — creates PDFs entirely in code
  • Platypus = page layout engine (like CSS for PDFs)
  • Flowables = building blocks: Paragraph, Table, Image, Spacer
  • SimpleDocTemplate = handles page size, margins, auto-pagination
  • No browser/template involved — pure programmatic PDF layout

Security

  • CSRF protection — Django's middleware blocks cross-site form submissions
  • Session auth — each request validated via server-side session
  • File upload validation — only image types accepted for profile photo
  • Django admin — role-separated, only superusers can access
  • ORM — all DB queries parameterized, SQL injection prevented

📸 Attendance Management System ML · Computer Vision

Face Recognition Attendance — MTCNN + FaceNet + SVM

Python · TensorFlow · PyTorch · OpenCV · MTCNN · FaceNet · SVM · Tkinter · openpyxl

FaceNetMTCNNSVMOpenCV

What it is: A desktop application that uses your webcam to automatically identify students' faces and mark their attendance in an Excel file. Works under varied lighting conditions. Built a full ML pipeline from dataset collection through training to real-time inference.

ML Pipeline Architecture

PHASE 1 — DATASET CREATION [Tkinter GUI] | Admin enters student name/ID v [OpenCV — webcam capture] | Captures ~100 face images per student v [MTCNN — face detection] | Detects face bounding box in each frame | Crops + aligns face to 160×160 pixels v [Saved to disk: /dataset/{student_name}/face_001.jpg ...] PHASE 2 — MODEL TRAINING [For each saved face image] | v [FaceNet — pretrained CNN (TensorFlow/PyTorch)] | Encodes face → 128-dimensional embedding vector | Same person → embeddings cluster close together | Different people → embeddings far apart v [SVM Classifier trained on embeddings] | Input: 128-d embedding vector | Output: student label | Trained on all students' embeddings v [Model saved: svm_model.pkl] PHASE 3 — REAL-TIME RECOGNITION [Webcam frame] → MTCNN detects face → crops it → FaceNet generates embedding → SVM predicts: "Ajay Thota" (confidence %) → If confidence > threshold: mark attendance → Write name + timestamp → Excel file (openpyxl)

End-to-End Flow

1
Dataset creation — Tkinter GUI collects student name → OpenCV opens webcam → captures ~100 images while student moves slightly → MTCNN finds and crops face in each image → saved to /dataset/StudentName/.
2
FaceNet embedding — each saved face image fed into the pretrained FaceNet model. FaceNet outputs a 128-dimensional vector (embedding) — a numerical fingerprint of that face. Same person always produces similar vectors.
3
SVM training — all embeddings labeled with student names. SVC(kernel='rbf') from scikit-learn trained to classify embedding vectors to identities. This is fast to train because FaceNet already did the heavy lifting.
4
Real-time recognition — webcam streams frames → MTCNN detects face box → crops to 160×160 → FaceNet generates embedding → SVM predicts name + confidence score → if confidence above threshold (e.g. 0.7), accepted as valid identification.
5
Attendance recording — confirmed identity + current timestamp written to Excel file via openpyxl. Duplicate logic prevents same student being marked twice in one session.

MTCNN — What & Why

  • Multi-task Cascaded CNN — 3 networks in sequence
  • Stage 1 (P-Net): fast scan for candidate face regions
  • Stage 2 (R-Net): refines candidates, rejects false positives
  • Stage 3 (O-Net): precise bounding box + facial landmarks
  • Why: accurate under varied lighting, angles, partial occlusion

FaceNet — What & Why

  • Pretrained deep CNN (Inception-ResNet architecture)
  • Maps face → 128-dimensional Euclidean space
  • Trained with Triplet Loss: same person closer, different people farther
  • Why pretrained: would need millions of faces to train from scratch — we fine-tune/use as feature extractor
  • SVM on top = fast to retrain when adding new students
💬 Recruiter talking point: "I split the ML work deliberately — FaceNet handles the hard problem (face representation) as a pretrained model, and SVM handles classification. This means adding a new student just requires generating their embeddings and retraining the SVM in seconds, not retraining a full neural network."
